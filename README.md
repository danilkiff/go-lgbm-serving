# go-lgbm-serving

[![CI](https://github.com/danilkiff/go-lgbm-serving/actions/workflows/ci.yml/badge.svg)](https://github.com/danilkiff/go-lgbm-serving/actions/workflows/ci.yml)

Подача обученной в Python модели **LightGBM** из **Go** через cgo, с **нативными
кодами причин SHAP** и harness **численного паритета**.

Это исследовательский proof-of-concept, не фреймворк. Он отвечает на один вопрос
конкретно, кодом и измерениями:

> Может ли Go подавать древесную ансамблевую модель, обученную в Python,
> воспроизводить её оценки и выдавать пригодные для регулятора коды причин на
> каждое решение - с задержкой и конкуренцией, нужными скореру платёжного фрода?

Полный рационал - цель, метод, рассмотренные альтернативы и измеренный результат
(включая то, почему нативный SHAP достижим для LightGBM/XGBoost, но не для
CatBoost/ONNX) - в [docs/DESIGN.md](docs/DESIGN.md); этот README - сводка.

## Что доказано и что нет

**Доказано** (инженерия, путь подачи):

- Go (cgo -> C API LightGBM) воспроизводит выход Python-модели.
- Нативные вклады **SHAP** идут прямо из того же предиктора
  (`C_API_PREDICT_CONTRIB`), а не из повторной реализации.
- harness паритета ловит дрейф оценки, SHAP и порядка кодов причин в CI.

**Не доказано**: ничего о *качестве детекции фрода*. Данные по умолчанию
синтетические (несбалансированные табличные). Это доказательство **сантехники и
корректности**, намеренно отделённое от эффективности модели. Реальный открытый
датасет подключается адаптером (`train.py --dataset csv`).

## Результат: паритет на одной сборке битоточный

На одной машине и одной сборке `lib_lightgbm` Go совпадает с Python точно на
holdout из 4000 строк:

| Проверка | Результат |
| --- | --- |
| Сырая маржа (Go против Python `predict(raw_score=True)`) | maxD = `0.000e+00` |
| Вклады SHAP (Go против Python `predict(pred_contrib=True)`) | maxD = `0.000e+00` |
| Инвариант sum(contrib) = сырая маржа | maxD = `1.5e-14` |
| Несовпадения порядка топ-3 кодов причин | `0 / 4000` |
| Смены решения (знак сырой маржи) | `0` |

**Кросс-платформенно, измерено** (macOS arm64 против Linux x86_64, тот же
`model.txt`, LightGBM 4.6.0, `num_threads=1`, float64): сырая маржа **битоточно
идентична** (maxD = `0`, `0/4000` смен решения); вклады SHAP расходятся лишь в
последнем бите (maxD = `1.1e-15`), порядок топ-3 кодов причин идентичен. При
контролируемых условиях - зафиксированная версия, один поток, float64, CPU -
детерминированный инференс фиксированной модели битостабилен между архитектурами,
далеко ниже гранулярности решения. (Это не противоречит предупреждению LightGBM
про кросс-окружение, которое касается *обучения*, порядка редукции в многопоточном
OpenMP, разных версий/компиляторов, float32 или GPU, но не детерминированного
инференса фиксированной модели на CPU.) Воспроизвести: `make xparity-dump` на
каждом хосте, затем
`uv run python xparity.py testdata/pred.darwin-arm64.csv testdata/pred.linux-amd64.csv`.

## Архитектура

- Сторона Python (`python/train.py`) обучает LightGBM и выгружает `model.txt`,
  матрицу holdout и эталонные предсказания + SHAP в `testdata/`.
- Сторона Go (`lgbm/`) загружает тот же `model.txt` через C API.
- **Приём для тесного паритета**: cgo линкуется с той же `lib_lightgbm`, что лежит
  внутри uv-venv Python. Go и Python поэтому исполняют один и тот же нативный
  предиктор - паритет настолько точен, насколько позволяет платформа.
- Прототипы C-ABI объявлены **вручную** в преамбуле cgo; мы *не* делаем
  `#include <LightGBM/c_api.h>` (он тянет C++/Arrow, которые cgo не компилирует).
  C API - стабильная поверхность `extern "C"`; версию либы фиксируем.

## Быстрый старт

Требуется: Go >= 1.24, [uv](https://docs.astral.sh/uv/) и рантайм OpenMP
(macOS: `brew install libomp`; Linux: `libgomp`, обычно есть).

```sh
make data    # python/train.py -> testdata/ (модель + holdout + эталон)
make test    # паритет + юнит-тесты
make bench   # задержка / пропускная способность
```

`make print-env` проверяет путь к нативной библиотеке, с которой линкуется cgo,
если сборка её не находит.

## Нативные грабли (реальная цена cgo + нативной либы)

- **Рантайм OpenMP**: `lib_lightgbm` линкует `@rpath/libomp.dylib`. Без него
  библиотека не сделает `dlopen`. Директива `#cgo darwin` в `lgbm/lgbm.go`
  добавляет rpath к brew libomp на macOS.
- **Пиннинг потоков**: предсказание передаёт `num_threads=1`. Порядок редукции
  float в многопоточном OpenMP - задокументированный источник недетерминизма
  между запусками; параллелизм держим на уровне Go (пул Booster), а не внутри
  нативного вызова predict.
- **Минимальная версия**: однострочный predict в C API не был потокобезопасен в
  LightGBM 3.0.0-3.1.1 (он ключевал внутренний буфер по `omp_get_thread_num()`,
  который для не-OpenMP вызывающих потоков, то есть потоков Go, возвращает 0,
  давая молчаливую гонку). Исправлено в
  [#3771](https://github.com/microsoft/LightGBM/pull/3771) (>= v3.2.0). Этот
  репозиторий фиксирует LightGBM 4.x.

## Численный паритет

`deterministic=true, force_row_wise=true, num_threads=1` - задокументированный
минимум для воспроизводимых результатов LightGBM, и обучение его использует.
Битоточный паритет держится на **одной сборке либы и платформе**. Между
компиляторами / ОС / архитектурами CPU сам LightGBM заявляет, что результаты
"ожидаемо различаются"
([docs](https://lightgbm.readthedocs.io/en/latest/Parameters.html)), но это
предупреждение про *обучение*, порядок редукции в многопоточном OpenMP, разнобой
версий, float32 и GPU. Кросс-платформенная задача здесь *измеряет* остаток, а не
предполагает его: для детерминированного однопоточного float64-**инференса**
фиксированной модели сырая маржа вышла битоточно идентичной arm64 против x86_64,
SHAP разошёлся лишь на `1.1e-15`. harness всё равно проверяет не только отклонение
оценки, но и **смены решения** и **устойчивость топ-K кодов причин**, поэтому
любой будущий дрейф (смена версии, числа потоков, путь float32) будет пойман, а не
принят на веру.

## Производительность

Задержка одиночной строки и пропускная способность пула (`make bench`; Apple
Silicon arm64, 12 потоков; синтетическая модель 30 признаков / 200 деревьев -
запустите свою на своей форме):

| Бенчмарк | нс/оп | Примечание |
| --- | ---: | --- |
| `PredictRaw` (1 горутина) | ~11700 | обычный скоринг |
| `PredictContrib` (1 горутина) | ~462000 | нативный SHAP - **примерно в 40 раз дороже скоринга** |
| `PredictRaw` через `Pool` (12 хэндлов, параллельно) | ~1410 | ~8x пропускной, без гонок |

Стоимость SHAP в 40 раз - количественный аргумент за архитектуру: **скорить каждую
транзакцию дёшево, считать коды причин вне горячего пути и только для отклонений**
(adverse action их требует, одобрения нет). Пул даёт настоящий параллелизм, потому
что каждая горутина берёт свой хэндл, обходя сериализованный/общий путь predict
(см. "Конкуренция" через #3751/#3771). Тест конкуренции гоняет 32 тысячи
предсказаний через пул под `-race` без расхождений.

## Реальные данные (опционально)

Доказательство по умолчанию работает на синтетике намеренно - так сантехника
подачи отделена от любого вопроса о самой модели. Чтобы прогнать тот же harness на
реальном публичном датасете фрода, используйте CSV-адаптер (`train.py --dataset csv`).

Пример - ULB Credit Card Fraud (Kaggle `mlg-ulb/creditcardfraud`: 284807
транзакций, 30 числовых признаков, ~0.17% положительных; Database Contents
License, здесь не хранится):

```sh
make data-creditcard   # скачать через kaggle CLI, затем обучить (python/fetch_creditcard.sh)
make test              # harness паритета теперь работает на модели из реальных данных
```

Это по-прежнему доказывает **путь подачи и корректность кодов причин**, а не
качество детекции фрода - адаптер лишь подменяет данные за тем же доказательством.
Подходит любой полностью числовой CSV: `--dataset csv --input <path> --target <column>`.

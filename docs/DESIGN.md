# Дизайн: цель, метод, альтернативы, результат

Фундаментальное требование: код причины на отклонение - точная атрибуция того же
предиктора, что принял решение, а не post-hoc приближение. Оно форсирует разом два
свойства:

- точная атрибуция - SHAP contributions из того же предиктора, что скорит вход;
- численный паритет Go-инференса с эталоном на Python - он и доказывает "тот же
  предиктор".

Остальной дизайн выводится отсюда.

## Метод и метрики

В этом решении cgo линкует тот же бинарник `lib_lightgbm` из venv Python (директива [`#cgo`](../serving/lgbm/lgbm.go)). Go и Python исполняют один и тот же нативный предиктор, и паритет ограничен только платформой.

Что проверяется:

| Критерий | Измерение | Где обеспечено |
| --- | --- | --- |
| Воспроизводит оценку | отклонение raw margin и смены решения | [`TestParityRaw`](../serving/lgbm/parity_test.go) |
| Нативный SHAP корректен | поэлементное отклонение от Python `predict(pred_contrib=True)` | [`TestParityContrib`](../serving/lgbm/parity_test.go) |
| SHAP внутренне согласован | инвариант sum(contrib) = raw margin | [`parity_test.go`](../serving/lgbm/parity_test.go) |
| Код причины устойчив | порядок топ-K признаков совпадает с эталоном | [`parity_test.go`](../serving/lgbm/parity_test.go) |
| Переносимо между платформами | дамп на каждой ОС, затем сравнение | [`cmd/dump`](../serving/cmd/dump/main.go) + [`xparity.py`](../training/xparity.py) |
| Конкуренция корректна | 64 горутины x 500 предсказаний под `-race` против однопоточного эталона | [`TestPoolConcurrentConsistency`](../serving/lgbm/pool_test.go) |
| Задержка / пропускная способность | оценка против contrib против пула | [`bench_test.go`](../serving/lgbm/bench_test.go) |

## Данные и признаки

Домен: сессионный фрод / RBA. Источник - [датасет RBA](https://www.kaggle.com/datasets/dasgroup/rba-dataset) (Wiefling et al.): ~31.3M попыток входа, 3.3M пользователей, крупный SSO в Норвегии. Данные синтезированы (правдоподобны, но искусственны, не для боевых IDS), поэтому репозиторий доказывает только путь инференса и корректность кодов причин, но не
эффективность детекции.

Цель: `Is Attack IP` (вход с атакующего IP), ~9.9% положительных - обучаемый сигнал
сессионного фрода. `Is Account Takeover` на порядки реже (флаг `--target ato`, но
мерить качество почти не на чем).

Признаки: числовые, поведенческие (RBA-новизна), считаются на пользователя в
хронологическом порядке - используют только прошлое, без утечки метки: время с
прошлого входа, число прошлых входов, час и день недели, успех входа, RTT, новизна
страны / города / ASN / ОС / браузера / устройства.

Глобально-частотные признаки страны и ASN выброшены: метка `Is Attack IP` выводится из
IP, страна и ASN - тоже, и их глобальная частота читает метку почти напрямую - 97.7%
gain, ROC-AUC 0.89 против 0.70 на поведенческих признаках. Это утечка, чьи коды причин
для adverse-action бессмысленны; без неё (0.70) коды объяснимы.

Режим оценки: случайный сплит оптимистичен. Временной (обучение до 85-го перцентиля
времени, тест после) даёт ROC-AUC 0.68 против 0.70 при дрейфе доли атак 0.096 -> 0.118.
Абсолютная PR-AUC между сплитами несравнима - она растёт с base rate (0.200 у временного
против 0.187 у случайного при худшей модели); сравнима lift = PR-AUC/base: 1.69x против
1.90x.

Цифры качества и утечки воспроизводимы: `make -C training validate` пишет
[validation.json](../training/results/validation.json), ноутбук
[analysis.ipynb](../training/results/analysis.ipynb) сводит их с утверждениями
README/DESIGN программно.

## Выбор: семейство модели

> Требуется точная атрибуция на табличных несбалансированных данных.

- нейросеть + post-hoc атрибуция (KernelSHAP / интегрированные градиенты): атрибуция
  приближённая и сэмплирующая, дороже в защите, и на табличных данных бустинг её
  обыгрывает ([Grinsztajn et al., 2022](https://arxiv.org/abs/2207.08815));
- GLM / скоркарта: коэффициенты и есть коды причин, но слабее на нелинейных
  взаимодействиях;
- градиентный бустинг деревьев + TreeSHAP: TreeSHAP точен, не приближение
  ([Lundberg et al.](https://arxiv.org/abs/1802.03888)), и быстр. Выбрано.

## Выбор: путь инференса в Go

> Точный нативный код причины на момент инференса возвращает только тот C-ABI, что сам отдаёт contributions.

| Путь инференса | Нативный SHAP на инференсе | Почему | Паритет |
| --- | :---: | --- | --- |
| LightGBM C API (cgo) | да | `C_API_PREDICT_CONTRIB` на публичных predict-функциях | битово точный - линкует ту же `lib_lightgbm` |
| XGBoost C API | да | маска SHAP-вывода `pred_contribs` | возможен, живого биндинга на Go нет |
| CatBoost `ModelCalcer` (C applier) | нет | applier только предсказывает; SHAP только в Python/CLI API | - |
| ONNX Runtime (cgo) | нет | нет SHAP-оператора ([onnxruntime#1176](https://github.com/microsoft/onnxruntime/issues/1176)); граф тензор-в/тензор-из | + конвертация = вторая поверхность расхождения |
| чистый Go [`leaves`](https://github.com/dmitryikh/leaves) | нет | только предсказание; реализует инференс заново | нет - вносит своё float-расхождение |

LightGBM и XGBoost отдают contributions на инференсе; CatBoost-applier и ONNX - нет.
Чистая реализация на Go проигрывает дважды: считает только предсказание, без
contributions, и переписывает инференс заново, внося своё float-расхождение.

Выбран LightGBM поверх XGBoost: чистый `C_API_PREDICT_CONTRIB`, линковка одного бинарника даёт битово точный паритет бесплатно и на сильной табличной модели.

## Выбор: примитивы конкуренции

Общий `Booster` за мьютексом сериализует предсказание; общий без мьютекса даёт
молчаливую гонку - C-API LightGBM 3.0.0-3.1.1 ключевал буфер по `omp_get_thread_num()`,
который для не-OpenMP-потоков Go возвращает 0 ([#3751](https://github.com/microsoft/LightGBM/issues/3751)).

Выбран пул независимых хэндлов, по одному на вызов, с одним нативным потоком на
предсказание (`num_threads=1` заодно убирает недетерминизм float-редукции OpenMP). Минимальная версия
>= 3.2.0 ([#3771](https://github.com/microsoft/LightGBM/pull/3771)); репозиторий
фиксирует 4.x.

## Результат

Доказано, трудная часть: код причины точен и доказуемо выводится из инференса, устойчивость порядка топ-K проверяется в CI. Это суть объяснимости
решений.

Построено: конвейер decline->explain - SHAP вне горячего пути, только для
отклонений, асинхронно, в одном процессе за интерфейсами `Queue`/`Store`
(`pipeline/` + `cmd/scorer`). `TestHotPathIsolation` держит p99 `/score` далеко ниже
одного SHAP при насыщенной очереди.

Контракт объяснений - best-effort с видимой потерей: ограниченная очередь может
отбросить событие, и тогда вызывающий узнаёт это сразу (`explain_queued=false` в
ответе `/score`), оператор - из `/metrics` (`queue_dropped`) и dead-letter.
Согласованность объяснения с решением обеспечена по построению - margin и
contributions считает один пул из одних байт - и закреплена паритетным и e2e
тестами; runtime-сверки нет намеренно, чтобы в однопроцессной топологии не
добавлять новый путь потери объяснения. В причины отклонения отбираются
только contributions, толкавшие к нему; факторы в пользу одобрения причиной
отклонения не бывают. Строгая гарантия "объяснение на каждое отклонение"
требует durable-очереди с подтверждением до ответа - адаптер за тем же
интерфейсом `Queue`, вне охвата вместе с сетевым развёртыванием.

Вне охвата: утверждённая таксономия adverse-action кодов и текст уведомления
(механизм есть - `reasoncode.Catalog` + `codes.json`, утверждённого набора кодов
нет), развёрнутый сетевой сервис, нагрузочный тест против SLA, эффективность
детекции (модель скромная и не тюнингованная - по дизайну доказывается путь, не
качество).

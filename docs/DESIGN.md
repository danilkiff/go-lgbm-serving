# Дизайн: цель, метод, альтернативы, результат

Документ формулирует один вопрос, на который отвечает репозиторий, взвешенные
ради ответа альтернативы и измеренный результат. Это зонтичный рационал; отдельные
локальные решения выводятся из комментариев непосредственно в коде, а
[README](../README.md) даёт сводку.

## Цель

> Может ли Go подавать древесную ансамблевую модель сессионного фрода, обученную в
> Python, воспроизводить её оценки и выдавать объяснимые коды причин
> на каждое решение - с задержкой и конкуренцией, нужными скореру входов
> (risk-based authentication)?

Вопрос распадается на четыре проверяемых подутверждения:

1. **Подать** обученный в Python древесный ансамбль из процесса Go.
2. **Воспроизвести его оценки** численно, вплоть до гранулярности решения.
3. **Выдавать коды причин на каждое решение**, *точные* и *доказуемо идентичные*
   объяснению за поданным решением (почему вход помечен рискованным).
4. Уложиться в **задержку и конкуренцию**, нужные высоконагруженному скореру входов.

Связующее ограничение - в подутверждении 3. Объяснение adverse-action (почему
вход отклонён или потребовал доп. фактор) должно быть *точной* атрибуцией *того же
предиктора*, что принял решение, а не приблизительной post-hoc рационализацией. Это
форсирует сразу два свойства: **точную атрибуцию** и **численный паритет** между
поданной Go-моделью и её эталоном на Python.

## Метод

Стержень - намеренно маленький приём: **линковать тот же бинарник `lib_lightgbm`,
что лежит внутри venv Python** (через [директиву `#cgo`](../serving/lgbm/lgbm.go)). Go и
Python исполняют *один и тот же нативный предиктор*. Паритет тогда настолько тесен,
насколько позволяет платформа, - не "две реализации согласны", а "одна реализация,
два вызывающих".

Поверх этого harness проверяет каждое подутверждение числом, а не утверждением:

| Подутверждение | Что измеряется | Где |
| --- | --- | --- |
| Воспроизводит оценку | отклонение сырой маржи **и смены решения** | [`TestParityRaw`](../serving/lgbm/parity_test.go) |
| Нативный SHAP корректен | поэлементное отклонение от Python `predict(pred_contrib=True)` | [`TestParityContrib`](../serving/lgbm/parity_test.go) |
| SHAP внутренне согласован | инвариант **sum(contrib) = сырая маржа** | [`parity_test.go`](../serving/lgbm/parity_test.go) |
| Код причины устойчив | **порядок топ-K признаков** совпадает с эталоном | [`parity_test.go`](../serving/lgbm/parity_test.go) |
| Держится между платформами | дамп на каждой ОС, затем diff | [`cmd/dump`](../serving/cmd/dump/main.go) + [`xparity.py`](../training/xparity.py) |
| Конкуренция корректна | 64 горутины x 500 предсказаний под `-race`, против однопоточного эталона | [`TestPoolConcurrentConsistency`](../serving/lgbm/pool_test.go) |
| Задержка / пропускная способность | оценка против contrib против пула | [`bench_test.go`](../serving/lgbm/bench_test.go) |

Паритет проверяется **не по среднему отклонению**, а по двум вещам, что реально
ломают продакшен, - **сменам решения** и **переупорядочиванию топ-K кодов причин**.

## Данные и признаки

Домен - сессионный фрод / RBA. Источник - [датасет RBA](https://www.kaggle.com/datasets/dasgroup/rba-dataset)
(Wiefling et al.): примерно 31.3M попыток входа, 3.3M пользователей, крупный SSO в
Норвегии. Данные **синтезированы** (авторы: правдоподобны, но искусственны, не для
боевых IDS), поэтому репозиторий честно доказывает *путь подачи и корректность
кодов причин*, а не эффективность детекции.

- **Цель**: `Is Attack IP` (вход с атакующего IP), примерно 9.9% положительных -
  обучаемый сигнал сессионного фрода. `Is Account Takeover` на порядки реже
  (доступен флагом `--target ato`, но мерить качество почти не на чем).
- **Признаки - числовые, поведенческие (RBA-новизна)**, считаются на пользователя
  в хронологическом порядке, поэтому используют только прошлое (без утечки метки):
  время с прошлого входа, число прошлых входов, час и день недели, успех входа,
  RTT, новизна страны / города / ASN / ОС / браузера / устройства для пользователя.
- **Выброшены глобально-частотные признаки страны/ASN.** Метка `Is Attack IP`
  выводится из IP; страна и ASN тоже из IP. Их глобальная частота почти напрямую
  читает метку: в эксперименте она давала 98.5% gain и ROC-AUC 0.89 - тривиальный
  шорткат, чьи коды причин бессмысленны для adverse-action. Без неё модель опирается
  на поведение пользователя (ROC-AUC 0.70), а коды причин становятся объяснимыми.
  Это и есть механизм RBA: "значения признаков отличаются от ранее наблюдавшихся у
  этого пользователя". Решение выводится из комментариев в [`train.py`](../training/train.py).

## Рассмотренные альтернативы

Стек выбран двухступенчатой воронкой. Второй фильтр - несущий.

### Фильтр 1: семейство модели

Требование: точная, дешёвая, защищаемая атрибуция на табличных несбалансированных
данных.

- **Нейросеть + post-hoc атрибуция** (KernelSHAP / интегрированные градиенты) -
  атрибуция *приблизительная и сэмплирующая*, дороже в защите и редко обыгрывает
  бустинг на табличных данных
  ([Grinsztajn et al., 2022](https://arxiv.org/abs/2207.08815)).
- **GLM / скоркарта** - коэффициенты *и есть* коды причин, но слабее на нелинейных
  взаимодействиях.
- **Градиентный бустинг деревьев + TreeSHAP** - TreeSHAP *точен*, не приближение
  ([Lundberg et al.](https://arxiv.org/abs/1802.03888)), и быстр. **Выбрано.**

### Фильтр 2: путь подачи в Go

Здесь решается "нативный SHAP - да/нет", и решается на уровне **инференсного C-ABI,
который линкует процесс Go**, а не Python-обёртки фреймворка.

| Путь подачи | Нативный SHAP на инференсе | Почему | Паритет |
| --- | :---: | --- | --- |
| **LightGBM C API** (cgo) | **да** | `C_API_PREDICT_CONTRIB` (=3) на публичных predict-функциях | **битоточный** - линкует ту же `lib_lightgbm` |
| **XGBoost C API** | **да** | маска SHAP-вывода `pred_contribs` | возможен, но живого биндинга на Go нет |
| **CatBoost** `ModelCalcer` (C applier) | **нет** | applier только предсказывает; SHAP только в Python/CLI API | - |
| **ONNX Runtime** (cgo) | **нет** | нет SHAP-оператора ([onnxruntime#1176](https://github.com/microsoft/onnxruntime/issues/1176)); граф тензор-в/тензор-из | + конвертация = вторая поверхность расхождения |
| чистый Go [`leaves`](https://github.com/dmitryikh/leaves) | нет | только предсказание; **реализует** инференс заново | нет - вносит своё float-расхождение |

**Несущий вывод:** точный нативный код причины на момент подачи достижим **только**
там, где сам инференсный C-ABI возвращает вклады, - **LightGBM и XGBoost да;
CatBoost-applier и ONNX нет.** Чистый Go-ридер проигрывает *дважды* - нет SHAP, и
его переписанный инференс ломает паритет. **LightGBM поверх XGBoost**: чистый путь
`C_API_PREDICT_CONTRIB`, приём с линковкой одного бинарника даёт битоточный паритет
даром, сильный табличный baseline.

### Конкуренция

Ни общий `Booster` за мьютексом (сериализует предсказание), ни общий без него (на
LightGBM 3.0.0-3.1.1 C API ключевал буфер по `omp_get_thread_num()`, который для
не-OpenMP потоков Go возвращает 0 - молчаливая гонка
[#3751](https://github.com/microsoft/LightGBM/issues/3751)). Вместо этого - **пул
независимых хэндлов**, по одному на вызов, с пиннингом потоков (`num_threads=1`,
что заодно убирает недетерминизм редукции OpenMP). Минимальная версия >= 3.2.0
([#3771](https://github.com/microsoft/LightGBM/pull/3771)); репозиторий фиксирует 4.x.

## Результат

Измерено на holdout из 50000 входов (LightGBM 4.6.0, float64):

| Подутверждение | Результат | Статус |
| --- | --- | :---: |
| Воспроизводит оценку (одна сборка) | сырая маржа maxD = `0.000e+00`, **0 смен решения** | доказано |
| Нативный SHAP корректен | contrib maxD = `0.000e+00` | доказано |
| SHAP согласован | sum(contrib) = raw, maxD = `1.51e-14` | доказано |
| Код причины устойчив | топ-3 несовпадений = `0 / 50000` | доказано |
| Кросс-платформенно (arm64 против x86_64) | сырая маржа **битоточно идентична** (maxD=0, 0 смен); SHAP расходится лишь в последнем бите (maxD = `7.8e-16`); топ-3 идентичны | измерено |
| Задержка | `PredictRaw` ~16 мкс; `PredictContrib` ~943 мкс (**примерно в 58 раз дороже скоринга**) | измерено |
| Конкуренция | Пул `PredictRaw` ~0.9 мкс, **~18x пропускной**, без гонок; 64x500 предсказаний, 0 расхождений | доказано |

Качество модели (вторично): на holdout (4963 положительных из 50000, base rate
9.9%) ROC-AUC = `0.703`, PR-AUC = `0.190` (база `0.099`). Сигнал поведенческий, ни
один признак не доминирует. Порог `margin>0` крайне консервативен (recall 0.4%);
рабочую точку задают по целевому FPR/recall (флаг `-threshold`).

**Ответ на центральный вопрос:** да - Go подаёт модель, воспроизводит её оценки
*точно* (битоточно на одной сборке, с измеренным остатком между архитектурами) и
выдаёт *точные нативные коды причин на каждое решение из того же предиктора*, с
устойчивостью топ-K, проверяемой в CI.

### Что "объяснимость решений" покрывает и что нет

- **Сделано** - трудная часть: объяснение *точное* и *доказуемо совпадает с
  поданным решением*, устойчивость порядка топ-K проверяется в CI. Это суть
  "объяснимости решений".
- **Построено** - конвейер decline->explain: SHAP вне горячего пути, только для
  отклонений; асинхронно, in-process за интерфейсами `Queue`/`Store`. Живёт в
  `pipeline/` + `cmd/scorer`; `TestHotPathIsolation` подтверждает, что p99 `/score`
  остаётся далеко ниже одного SHAP при насыщенной очереди.
- **Вне охвата** - валидированная таксономия adverse-action кодов и текст
  уведомления (механизм отображения есть - `reasoncode.Catalog` + `codes.json`, - но
  не утверждённый набор кодов), развёрнутый сетевой сервис, нагрузочный тест против
  SLA, и сама эффективность детекции (модель честная, но скромная и не тюнингованная
  - это по дизайну: доказывается путь, не качество).

Коротко: **"можно ли это сделать" отвечено "да" на доказанной механике; "развёрнуто
ли в продакшен" намеренно вне охвата.** Каждое "да" выше подкреплено числом.

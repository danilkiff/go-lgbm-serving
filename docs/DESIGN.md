# Дизайн: цель, метод, альтернативы, результат

Документ формулирует один вопрос, на который отвечает репозиторий, взвешенные
ради ответа альтернативы и измеренный результат. Это зонтичный рационал; отдельные
локальные решения выводятся из комментариев непосредственно в коде, а
[README](../README.md) даёт сводку.

## Цель

> Может ли Go подавать древесную ансамблевую модель, обученную в Python,
> воспроизводить её оценки и выдавать пригодные для регулятора коды причин на
> каждое решение - с задержкой и конкуренцией, нужными скореру платёжного фрода?

Вопрос распадается на четыре проверяемых подутверждения:

1. **Подать** обученный в Python древесный ансамбль из процесса Go.
2. **Воспроизвести его оценки** численно, вплоть до гранулярности решения.
3. **Выдавать коды причин на каждое решение**, *точные* и *доказуемо идентичные*
   объяснению за поданным решением.
4. Уложиться в **задержку и конкуренцию**, нужные высоконагруженному скореру фрода.

Связующее ограничение - в подутверждении 3. Объяснение adverse-action (почему
транзакция отклонена) должно быть *точной* атрибуцией *того же предиктора*, что
принял решение, а не приблизительной post-hoc рационализацией, которая может с ним
разойтись. Это требование форсирует сразу два свойства: **точную атрибуцию** и
**численный паритет** между поданной Go-моделью и её эталоном на Python. Всё ниже
следует из удержания этих двух вместе.

## Метод

Стержень - намеренно маленький приём: **линковать тот же бинарник `lib_lightgbm`,
что лежит внутри venv Python** (через [директиву `#cgo`](../lgbm/lgbm.go) в обёртке
cgo). Go (через cgo) и Python поэтому исполняют *один и тот же нативный предиктор*.
Паритет тогда настолько тесен, насколько позволяет платформа, - не "две реализации
согласны", а "одна реализация, два вызывающих".

Поверх этого harness проверяет каждое подутверждение числом, а не утверждением:

| Подутверждение | Что измеряется | Где |
| --- | --- | --- |
| Воспроизводит оценку | отклонение сырой маржи **и смены решения** (знак маржи) | [`TestParityRaw`](../lgbm/parity_test.go) |
| Нативный SHAP корректен | поэлементное отклонение от Python `predict(pred_contrib=True)` | [`TestParityContrib`](../lgbm/parity_test.go) |
| SHAP внутренне согласован | инвариант **sum(contrib) = сырая маржа** | [`parity_test.go`](../lgbm/parity_test.go) |
| Код причины устойчив | **порядок топ-K признаков** совпадает с эталоном | [`parity_test.go`](../lgbm/parity_test.go) |
| Держится между платформами | дамп на каждой ОС, затем diff | [`cmd/dump`](../cmd/dump/main.go) + [`xparity.py`](../python/xparity.py) |
| Конкуренция корректна | 64 горутины x 500 предсказаний под `-race`, против однопоточного эталона | [`TestPoolConcurrentConsistency`](../lgbm/pool_test.go) |
| Задержка / пропускная способность | оценка против contrib против пула | [`bench_test.go`](../lgbm/bench_test.go) |

Методологически важный выбор: паритет проверяется **не по среднему отклонению**, а
по двум вещам, что реально ломают продакшен, - **сменам решения** и
**переупорядочиванию топ-K кодов причин**. Кросс-платформенное расхождение
*измеряется*, а не списывается.

## Рассмотренные альтернативы

Стек выбран двухступенчатой воронкой. Второй фильтр - несущий.

### Фильтр 1: семейство модели

Требование: точная, дешёвая, защищаемая атрибуция на табличных несбалансированных
данных.

- **Нейросеть + post-hoc атрибуция** (KernelSHAP / интегрированные градиенты) -
  атрибуция *приблизительная и сэмплирующая*, дороже в защите и редко обыгрывает
  бустинг на табличных данных
  ([Grinsztajn et al., 2022](https://arxiv.org/abs/2207.08815)).
- **GLM / скоркарта** - коэффициенты *и есть* коды причин, но заметно слабее на
  нелинейных взаимодействиях, типичных для фрода.
- **Градиентный бустинг деревьев + TreeSHAP** - TreeSHAP *точен*, не приближение
  ([Lundberg et al.](https://arxiv.org/abs/1802.03888)), и быстр. **Выбрано.**

### Фильтр 2: путь подачи в Go

Здесь и решается "нативный SHAP - да/нет", и решается на уровне **инференсного
C-ABI, который линкует процесс Go**, а не Python-обёртки фреймворка. Фреймворк
может считать SHAP в Python и при этом *не* отдавать никаких вкладов через
C-applier, который вызвал бы сервис на Go.

| Путь подачи | Нативный SHAP на инференсе | Почему | Паритет |
| --- | :---: | --- | --- |
| **LightGBM C API** (cgo) | **да** | `C_API_PREDICT_CONTRIB` (=3) на публичных predict-функциях | **битоточный** - линкует ту же `lib_lightgbm` |
| **XGBoost C API** | **да** | маска SHAP-вывода `pred_contribs` | возможен, но живого биндинга на Go нет |
| **CatBoost** `ModelCalcer` (C applier) | **нет** | applier только предсказывает; SHAP живёт только в Python/CLI API | - |
| **ONNX Runtime** (cgo) | **нет** | нет SHAP-оператора ([onnxruntime#1176](https://github.com/microsoft/onnxruntime/issues/1176)); граф тензор-в/тензор-из, только оценка | + конвертация в ONNX = вторая поверхность расхождения |
| чистый Go [`leaves`](https://github.com/dmitryikh/leaves) | нет | только предсказание; **реализует** инференс заново | нет - вносит своё float-расхождение |
| Treelite / TL2cgen | нет | компилирует предиктор; нативного TreeSHAP из коробки нет | - |

**Несущий вывод:** точный нативный код причины на момент подачи достижим **только**
там, где сам инференсный C-ABI возвращает вклады, - **LightGBM и XGBoost да;
CatBoost-applier и ONNX нет.** SHAP в CatBoost так и не вышел из тяжёлой Python/C++
библиотеки; у ONNX такого оператора нет вовсе. Чистый Go-ридер проигрывает *дважды*
- нет SHAP, и его переписанный инференс ломает паритет.

**LightGBM поверх XGBoost**, потому что: (1) чистый путь `C_API_PREDICT_CONTRIB`;
(2) приём с линковкой одного бинарника даёт битоточный паритет даром; (3) сильный
табличный baseline без героического тюнинга. Операционная цена cgo + нативной
библиотеки (`.so`/`.dylib` для поставки, rpath OpenMP, отсутствие изоляции крэшей)
принята как *цена требования*, а не золочение, - это **минимальный** путь,
удовлетворяющий "точный SHAP и паритет".

### Конкуренция

Ни общий `Booster` за мьютексом (сериализует предсказание), ни общий `Booster` без
него - на LightGBM 3.0.0-3.1.1 C API ключевал внутренний буфер по
`omp_get_thread_num()`, который для не-OpenMP потоков Go возвращает 0, молчаливая
гонка ([#3751](https://github.com/microsoft/LightGBM/issues/3751)). Вместо этого -
**пул независимых хэндлов**, по одному на вызов, с пиннингом нативных потоков
(`num_threads=1`, что заодно убирает недетерминизм порядка редукции OpenMP).
Минимальная версия >= 3.2.0
([#3771](https://github.com/microsoft/LightGBM/pull/3771)); репозиторий фиксирует
4.x.

## Результат

Измерено на holdout из 4000 строк (LightGBM 4.6.0, `num_threads=1`, float64):

| Подутверждение | Результат | Статус |
| --- | --- | :---: |
| Воспроизводит оценку (одна сборка) | сырая маржа maxD = `0.000e+00`, **0 смен решения** | доказано |
| Нативный SHAP корректен | contrib maxD = `0.000e+00` | доказано |
| SHAP согласован | sum(contrib) = raw, maxD = `1.5e-14` | доказано |
| Код причины устойчив | топ-3 несовпадений = `0 / 4000` | доказано |
| Кросс-платформенно (arm64 против x86_64) | сырая маржа **битоточно идентична** (maxD=0, 0 смен); SHAP расходится лишь в последнем бите (maxD = `1.1e-15`); порядок топ-3 идентичен | измерено |
| Задержка | `PredictRaw` ~11.7 мкс; `PredictContrib` ~462 мкс (**примерно в 40 раз дороже скоринга**) | измерено |
| Конкуренция | Пул (12 хэндлов) `PredictRaw` ~1.41 мкс, **~8x пропускной**, без гонок; 64x500 предсказаний, 0 расхождений | доказано |

**Ответ на центральный вопрос:** да - Go подаёт модель, воспроизводит её оценки
*точно* (битоточно на одной сборке, с измеренным остатком между архитектурами) и
выдаёт *точные нативные коды причин на каждое решение из того же предиктора*, с
устойчивостью топ-K, проверяемой в CI. Подутверждения (1), (2) и *механизм* (3) -
твёрдое "да".

### Что "пригодность для регулятора" и "конкуренция скорера фрода" покрывают и что нет

Сформулировано точно, чтобы утверждение оставалось защищаемым перед валидацией
модели:

- **Сделано** - трудная часть: объяснение *точное* и *доказуемо совпадает с
  поданным решением* (не приблизительная post-hoc атрибуция), а устойчивость
  порядка топ-K проверяется в CI. Это и есть суть "пригодности для регулятора".
- **Построено** - пайплайн decline->explain. SHAP считается вне горячего пути и
  только для отклонений; топология асинхронная, in-process за интерфейсами
  `Queue`/`Store`. Живёт в `pipeline/` + `cmd/scorer` (`POST /score`,
  `GET /explain/{id}`, `GET /metrics`); `TestHotPathIsolation` подтверждает, что
  p99 `/score` остаётся далеко ниже одного SHAP при насыщенной очереди explain.
- **Вне охвата** - стандартизированная/валидированная таксономия кодов
  adverse-action и текст уведомления (механизм настраиваемого отображения есть -
  `reasoncode.Catalog`, - но не регуляторный набор кодов), развёрнутый сетевой
  сервис и нагрузочный тест QPS против SLA. Задержка на вызов и пропускная
  способность пула измерены на одной форме машины; сквозной SLA на боевой
  инфраструктуре - нет.

Коротко: **"можно ли это сделать" отвечено "да" на доказанной механике; "развёрнуто
ли в продакшен" намеренно вне охвата.** Такая рамка - позиция силы: каждое "да"
выше подкреплено числом.

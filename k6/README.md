# Нагрузочное тестирование

k6-нагрузка через REST по матрице 2x2 (хранилище x лимит), в двух режимах
объяснения:

|            | без лимита      | 1 CPU / 1 GB    |
|------------|-----------------|-----------------|
| in-memory  | `mem-unlimited` | `mem-limited`   |
| postgresql | `pg-unlimited`  | `pg-limited`    |

Лимит вешается только на контейнер `scorer` (`docker-compose.limit.yml`);
postgres держит свои ресурсы.

Режимы объяснения (`scorer -explain`):

- **async** (дефолт): `/score` скорит и для отклонений кладёт событие в очередь;
  SHAP считают 2 фоновых воркера вне горячего пути, результат сохраняется и
  отдаётся `GET /explain/{id}`. Горячий путь хранилище не трогает. Под нагрузкой,
  превышающей пропускную способность воркеров, очередь переполняется и объяснения
  **теряются**.
- **inline**: `/score` сам считает SHAP для КАЖДОГО решения (принятого и
  отклонённого) и сохраняет объяснение до ответа. Ни очереди, ни воркеров -
  потерять объяснение нечем. Цена: каждый `/score` платит полный SHAP.

`-store=postgres` - адаптер `pipeline.PgStore` за тем же интерфейсом `Store`, что
и `MemStore` (таблица `explanations`, id PK, reasons jsonb).

## Команды

```sh
bash k6/gen-rows.sh                              # rows.json + threshold (~50% отклонений)
bash k6/bench.sh                                 # async, 1000 rps x 30s
EXPLAIN=inline bash k6/bench.sh                  # inline
EXPLAIN=inline RATE=4000 bash k6/bench.sh        # inline под стрессом
```

Env: `EXPLAIN` (async|inline), `RATE` (POST /score в секунду), `DURATION`, `VUS`,
`MAX_VUS`, `WORKERS` (воркеров explain в async). Сводка печатается в конце, JSON -
в `k6/results/<perm>.json` (+ снимок `/metrics` в `<perm>.metrics.json`).
Архивы прогонов: `k6/results/{async,inline}-{1000,4000}rps/`.

Файлы: `Dockerfile.scorer` (cgo против lib_lightgbm из колеса pip),
`docker-compose.yml`, `docker-compose.limit.yml`, `score-explain.js`, `bench.sh`,
`gen-rows.sh` (вызывает `gen_rows.py`, `gen_threshold.py`), `summary.py` (сводка
прогонов).

## Результаты

Apple M4 Pro, Docker Desktop, go 1.26, lightgbm 4.6, postgres:16-alpine, 2 воркера
(async), очередь 1024, отклонений ~50%, латентности в мс. "Объяснений/с" =
сохранённых объяснений в секунду; "потеряно" = queue_dropped.

### async (объяснение вне горячего пути)

> Узкое место (4000 rps): pg-unlimited - запись explain (2 воркера x commit-RTT
> pg), pg-limited - CPU одного ядра scorer; mem-конфиги и весь 1000 rps не
> насыщены. Замеры CPU и поячейковый разбор - в конце раздела.

1000 rps (~500 отклонений/с) - поспевают все:

| perm          | /score p99 | GET p99 | explain wait p99 | объясн/с | found | потеряно |
|---------------|-----------:|--------:|-----------------:|---------:|------:|---------:|
| mem-unlimited |       1.21 |    1.18 |             6.09 |     2007 |  1.00 |        0 |
| mem-limited   |       1.28 |    1.48 |             6.14 |     2009 |  1.00 |        0 |
| pg-unlimited  |       0.63 |    0.78 |            10.13 |      500 |  1.00 |        0 |
| pg-limited    |       0.69 |    0.96 |            10.22 |      498 |  1.00 |        0 |

4000 rps (~2000 отклонений/с) - pg упирается в потолок записи:

| perm          | /score p99 | GET p99 | explain wait p99 | объясн/с | found | потеряно |
|---------------|-----------:|--------:|-----------------:|---------:|------:|---------:|
| mem-unlimited |       1.43 |    1.23 |             6.14 |     2007 |  1.00 |        0 |
| mem-limited   |       2.01 |    1.90 |             6.46 |     2009 |  1.00 |        0 |
| pg-unlimited  |       4.74 |    4.51 |            95.89 |      790 |  0.01 |    35848 |
| pg-limited    |      43.65 |    7.85 |            98.29 |      128 |  0.00 |    46964 |

Горячий путь быстр, но под пиком pg-воркеры не успевают: 60% (без лимита) и 92%
(1 CPU) отклонений **остаются без объяснения** - тихая потеря.

### inline (объяснение на горячем пути для всех)

> Узкое место (4000 rps): *-limited - CPU одного ядра scorer (SHAP для всех) ->
> backlog HTTP и сброс нагрузки; pg-unlimited - латентность commit pg на /score
> (не throughput); mem-unlimited и весь 1000 rps не насыщены. Замеры CPU и
> поячейковый разбор - в конце раздела.

1000 rps - поспевают все, объяснение готово сразу (хранится до ответа):

| perm          | /score p99 | GET p99 | объясн/с | found | потеряно |
|---------------|-----------:|--------:|---------:|------:|---------:|
| mem-unlimited |       1.30 |    0.69 |     1000 |  1.00 |        0 |
| mem-limited   |       1.38 |    0.74 |     1000 |  1.00 |        0 |
| pg-unlimited  |       3.44 |    0.63 |     1000 |  1.00 |        0 |
| pg-limited    |       3.05 |    0.69 |     1000 |  1.00 |        0 |

4000 rps - **ничего не теряется**, цена уходит в латентность и сброс нагрузки:

| perm          | /score p99 | объясн/с | found | потеряно |    принято из 4000/с |
|---------------|-----------:|---------:|------:|---------:|---------------------:|
| mem-unlimited |       1.45 |     4000 |  1.00 |        0 |                 4000 |
| mem-limited   |     545.05 |     3013 |  1.00 |        0 | **3013 (сброс 24%)** |
| pg-unlimited  |      29.67 |     4000 |  1.00 |        0 |                 4000 |
| pg-limited    |     444.23 |     2682 |  1.00 |        0 | **2682 (сброс 33%)** |

GET p99 в inline = explain wait p99 = доли мс: объяснение уже в хранилище к
моменту ответа, первый же опрос попадает. "Сброс" - доля запросов, которые
генератор не смог запустить (горячий путь насыщен), т.е. backpressure, а не потеря
объяснения: каждое обслуженное решение объяснено.

### Сравнение под стрессом (4000 rps): доставлено объяснений без потерь

| perm         | режим  | /score p99 | объясн/с |    потеряно |
|--------------|--------|-----------:|---------:|------------:|
| pg-unlimited | async  |       4.74 |      790 |   **35848** |
| pg-unlimited | inline |      29.67 |     4000 |           0 |
| mem-limited  | async  |       2.01 |     2009 |           0 |
| mem-limited  | inline |     545.05 |     3013 |           0 |

### Узкие места по тестам (с замером CPU)

Пиковый CPU контейнеров под стрессом (4000 rps, `docker stats` во время прогона).
scorer без лимита делит 12 ядер хоста; под лимитом ограничен 1 ядром, т.е. 100% =
одно ядро. `-` = postgres в тесте не участвует.

| тест                 | scorer CPU | pg CPU | узкое место (аргумент)                                                  |
|----------------------|-----------:|-------:|-------------------------------------------------------------------------|
| async mem-unlimited  |      ~105% |    -   | нет: ~1 ядро из 12, 0 потерь - предел задаёт предложенный RPS            |
| async mem-limited    |       ~88% |    -   | CPU 1 ядра у предела (~88%), но ещё поспевает - 0 потерь                 |
| async pg-unlimited   |      ~196% |  ~117% | запись explain: 2 воркера x commit-RTT pg; ни scorer (2/12), ни pg (1.2 ядра) не максятся |
| async pg-limited     |      ~103% |   ~85% | CPU scorer (1 ядро): проседает даже скоринг (<4000/с)                    |
| inline mem-unlimited |      ~150% |    -   | нет: ~1.5 ядра из 12, 0 сброса - предел задаёт RPS                       |
| inline mem-limited   |      ~102% |    -   | CPU scorer (1 ядро): чистый SHAP, перекрыть простоем нечем (запись mem бесплатна) |
| inline pg-unlimited  |      ~176% |   ~56% | латентность commit pg на /score (p99 ~34мс), не throughput: запас по ядрам и pg есть |
| inline pg-limited    |      ~103% |   ~40% | CPU scorer (1 ядро); pg лишь ~40% - стена в ядре, не в базе              |

Три механизма (всё при 1000 rps и оба `*-unlimited` на mem не насыщены - кратный
запас по CPU, потерь нет, предел задаёт предложенный RPS, а не система):

- **CPU одного ядра** (async pg-limited, inline mem-limited, inline pg-limited):
  scorer пришпилен к ~100% своего единственного ядра. У inline mem-limited это
  чистый SHAP - запись в память бесплатна, простоем перекрыть нечего; у inline
  pg-limited - SHAP плюс CPU драйвера pg, причём сама pg лишь ~40%, т.е. упёрто
  ядро, а не база; у async pg-limited ядро делят скоринг и 2 воркера. Симптом -
  /score p99 в сотни мс и сброс 24-33% (inline) либо просадка скоринга <4000/с
  (async).
- **Запись explain = 2 воркера x RTT pg** (async pg-unlimited): и scorer (2 ядра
  из 12), и pg (~1.2 ядра) недогружены - значит стена не в CPU и не в IOPS базы, а
  в числе воркеров. Каждый блокируется на ~2мс синхронного commit -> ~800-1000
  объяснений/с и потеря 28-36 тыс. Лечится воркерами/батчингом записи, а не дисками.
- **Латентность commit pg на горячем пути** (inline pg-unlimited): throughput
  держит 4000/с (запас по ядрам и pg есть), но каждый /score синхронно ждёт
  commit -> p99 ~34мс. Узкое место - задержка, не пропускная способность.

## Выводы

> Объяснение инференса на горячем пути повышает надёжность системы

Режим async буферизует SHAP в очереди и под пиком её переполняет - объяснения теряются молча
(found -> 0.01, до 92% потерь). inline убирает очередь как класс: объяснение
считается и сохраняется в самом запросе, поэтому потерять его нечем (found=1.00,
потеряно=0 во всех конфигурациях, на любой нагрузке).

**Контринтуитивно растёт и пропускная способность объяснений.** async упирает SHAP
в `WORKERS=2` (~790 объяснений/с для pg). inline параллелит SHAP по всем
`GOMAXPROCS` хэндлам горячего пула: pg-unlimited под стрессом выдаёт 4000
объяснений/с против 790 у async - в 5 раз больше, и ничего не теряя. При обилии
ядер inline одновременно надёжнее и производительнее по объяснениям.

**Цена - латентность /score и вдвое больше работы CPU.** Каждый /score платит
SHAP: p99 mem-unlimited почти не меняется (ядра поглощают), pg-unlimited 4.7 ->
29.7мс, а под 1 CPU горячий путь насыщается (p99 440-545мс, 24-33% нагрузки
сброшено), потому что inline считает SHAP для ВСЕХ решений, а не только отклонений
- вдвое больше SHAP, чем у async. Под жёстким лимитом CPU inline держит примерно
вдвое меньший RPS.

**Режим отказа становится явным.** async под перегрузкой возвращает
быстрые решения, но без объяснений (тихая потеря). inline под перегрузкой
замедляет и сбрасывает запросы (видимый backpressure) - но что обслужено, то
объяснено. Для объяснимости отказов это и есть искомая надёжность.

Выбор режима - это выбор, что важнее: async, когда нужен максимально быстрый и
ёмкий /score, а объяснения допустимо терять под пиком; inline, когда у каждого
решения обязано быть объяснение, и за это можно платить латентностью и CPU.

## Замечания

- Замеры на Docker Desktop (Mac, VM): сеть ~1мс маскирует разницу хранилищ
  на единичной операции. Потолок записи, параллелизм SHAP и контеншн за CPU от
  среды не зависят - выводы переносимы.
- inline под устойчивой перегрузкой растит неограниченный backlog в HTTP-слое
  (нет встроенного сброса) - в проде нужен ограничитель конкурентности / 429.
  inline также связывает горячий путь с хранилищем: латентность pg теперь в /score.
- Потолок async завязан на `WORKERS=2`; больше воркеров - выше потолок (но всё
  ещё ниже параллелизма inline по GOMAXPROCS).
- `--cpus=1` -> квота cgroup -> Go 1.26 ставит `GOMAXPROCS=1` (1 хэндл горячего
  пула, виден в логе старта).
